What is version control?

Version control is a system that records changes to a file or set of files over time so you can have previous versions of the file available in the future. VCS lets you revert back to previous file or project versions, compare changes to files/projects over time, see who last modified something, and more. Git is the most commonly used VCS now. Before 2002, Linux kernel, a large open source software project that provided file maintenance through patches and archived files. In 2002, Linux kernel began using a free VCS called BitKeeper. Tensions between the community that used BitKeeper and the corporation that created it heightened, causing the creators to make it a paid only service. In 2005, Git was created to be a better, faster, and free version of BitKeeper.

Why is it important?

Version control is an important tool to many developers because it stores all versions of files and projects, allows you to restore previous files and projects, tracks changes, allows for collaboration, and keeps backups. When saving files to a local client, you may overwrite previous versions of files, or when working as a team, share multiple versions of the same file back and forth, leaving room for errors to occur. VCS save all versions of files, so even if you accidentally lose a file or overwrite someone elseâ€™s changes, you have every version of the file saved, and can always revert back to a previous version. You can also track all changes made to a file by comparing the current version to any previous version, which can be helpful in debugging. VCS also allows every member of a team to work on any file at any time, and when saved, will merge all the changes into a common version. Each team member also has their own personal mirrored copy of the project, meaning there are multiple backups of the same project available.

Three types of VCS

There are three types of version control systems: local, centralized, and distributed. Local VCS copy files from your computer to a directory. This, however, can leave you vulnerable to errors because you can end up using the wrong directory or copying the wrong files. Centralized VCS is a single server that contains all the versioned files. This makes group work easy, and allows for greater administrative oversight because you can track what changes are made by who. However, if the main server goes down, no one can work in the project. Distributed VCS allow clients to perfectly mirror a repository, so every clone is a full backup of the data. This allows you to work whenever, wherever, regardless of if you are connected to the internet or not.

How does it work? r

Git works by saving files as a stream of snapshots; every time a file is saved, a picture of the new data is taken and stored. Local databases are used, so old versions are easy to pull up, and you can work offline until a connection is made so you can upload the newest version. Once you commit a snapshot, it is hard to lose. There are three main states a file can be in: modified, staged, and committed. A modified file is a file that has been changed, but not saved. In order to save a file, you must stage it first; which is when a modified file in its current version is marked to be committed. A committed file has been successfully saved and stored in the local database.

Git basics

Git is the most popular form of a VCS. In order to create a Git project, or a repository, you can copy over local directory files (using the command cd /c/user/my_project) or clone and existing repository (using command git clone ). To start tracking a file, you must use the command git addme. In order to check the status of a file, use the command git status. Commit files by using git commit, and skip the staging phase by using git commit -a. To see what changes have been made, use git diff. The command git rm will remove a file, while git mv file_from file_to will change the name of a file.
